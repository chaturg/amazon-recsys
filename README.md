# Amazon Product Recommender — Two-Stage Retrieval System

A production-style two-stage retrieval pipeline trained on the Amazon Reviews 2023 dataset
(Tools & Home Improvement, 26M reviews). Built as a portfolio project to demonstrate
end-to-end ML systems design for applied ML engineering roles.

**Live demo:** [HuggingFace Spaces — chaturg/amazon-recsys-demo](https://huggingface.co/spaces/chaturg/amazon-recsys-demo)
**Dataset:** [chaturg/amazon-recsys-dataset](https://huggingface.co/datasets/chaturg/amazon-recsys-dataset)
**Model:** [chaturg/amazon-recsys-cross-encoder](https://huggingface.co/chaturg/amazon-recsys-cross-encoder)

## Architecture

```
User purchase history
        │
        ▼
┌───────────────────┐     ┌──────────────────────────────────┐
│   User Tower      │     │   Item Tower                     │
│   (MLP, 7 feats)  │     │   (MLP, 384-dim title embeddings)│
│                   │     │   all-MiniLM-L6-v2 → item vecs   │
└────────┬──────────┘     └──────────────┬───────────────────┘
         │                               │
         │         128-dim embeddings    │
         └──────────────┬────────────────┘
                        ▼
              FAISS IVF Index
              (157k items, nlist=100, nprobe=10–20)
              Top-100 candidates in ~0.3s
                        │
                        ▼
          Cross-Encoder Re-Ranker
          (fine-tuned ms-marco-MiniLM-L-6-v2)
          [CLS] user_history [SEP] item_title [SEP]
          Pointwise cross-entropy · top-10 in ~0.5s
                        │
                        ▼
              Top-10 Recommendations
              Claude Haiku LLM judge scores each 0–5
```

**Why two stages?** The bi-encoder + FAISS retrieval is fast but approximate — it can
scan 157k items in milliseconds because both sides are pre-encoded independently.
The cross-encoder is slower but sees the full interaction between user context and item
text, giving much higher-quality re-ranking. This is the standard production pattern
at Google, Pinterest, Airbnb, and Spotify.

## Models Used

| Model | Role | Training |
|---|---|---|
| `sentence-transformers/all-MiniLM-L6-v2` | Encodes 157k item titles → 384-dim vectors | Pretrained, not fine-tuned |
| **TwoTowerModel** (custom PyTorch) | Learns user taste from purchase history | Trained from scratch on Amazon Reviews |
| `cross-encoder/ms-marco-MiniLM-L-6-v2` | Re-ranks top-100 candidates to top-10 | Fine-tuned on domain-specific pairs |
| Claude Haiku (Anthropic API) | LLM judge — scores recommendations 0–5 | Not trained — used for evaluation only |

## Data Pipeline

**Source:** Amazon Reviews 2023 — Tools & Home Improvement (McAuley Lab).
License: non-commercial research use only.

**Cold-start filtering:** Minimum 5 interactions per user, 10 per item.
Three iterative passes (items filtered first, then users) to resolve the circular
dependency — removing cold-start items creates cold-start users.

**Silver label construction:** Raw star ratings are user-biased — a 3★ from a harsh
rater carries more positive signal than a 3★ from someone who only leaves 5s.
Silver labels correct for this by combining four signals:

| Signal | Weight | Rationale |
|---|---|---|
| Per-user z-scored rating | 0.50 | Corrects individual rating bias — the dominant signal |
| Helpfulness votes | 0.20 | Community validation; log-scaled to dampen viral outliers |
| Verified purchase | 0.15 | Higher signal quality than unverified reviews |
| Review length | 0.15 | Longer reviews indicate stronger engagement |

**Train/val/test split:** Temporal leave-one-out per user — last interaction → test,
second-to-last → val, all prior → train. Every user appears in all three splits.
No future interactions leak into training.

## Training — 3 Configurations

Each configuration adds a more sophisticated negative sampling strategy:

| Config | Negative Sampling | Purpose |
|---|---|---|
| Config 1 (Baseline) | Random negatives | Establishes floor — random items are easy negatives |
| Config 2 | In-batch negatives | Other users' items in same batch — moderate difficulty |
| Config 3 (Full system) | ANN hard negatives | FAISS top-100 candidates as negatives — items that look similar but aren't relevant |

Hard negatives are the key improvement — they force the model to distinguish between
items that are close in embedding space but not actually relevant to the user.

**Cross-encoder training:** Fine-tuned on Config 3 retrieval pairs using pointwise
cross-entropy against silver labels (not pairwise hinge). Silver labels are continuous
per-item scores — cross-entropy matches the base model's MS MARCO pre-training
distribution and treats each (query, item, label) triple independently.

### A note on offline metrics

This system is trained on **explicit feedback** — written reviews tell us what users
liked in the past, not what they intend to buy next. Offline purchase prediction metrics
(Recall@100, NDCG@10) measure whether the model can retrieve one specific held-out
review from 157k items. This is an extremely hard target that understates real-world
recommendation quality for two reasons:

1. With sparse review data (median ~8 reviews per user), the "last review" is partly
   random — there is no strong temporal signal to predict.
2. Reviews are retrospective. Recommendations are prospective. These are different tasks.

**In production, this system would be evaluated via online A/B testing** on click-through
rate and add-to-cart rate. The Claude Haiku LLM judge scores shown in the demo
(relevance of recommendations given user history) are a more appropriate offline proxy
than single-item purchase prediction metrics.

The architecture, label engineering, and negative sampling progression are the primary
demonstrations of ML engineering judgment here — not the offline metric values.

## Cold Start Strategy

| Scenario | Detection | Response |
|---|---|---|
| Net-new user | 0 interactions | Popularity fallback by category — most-reviewed items shown |
| Sparse user | < 5 interactions | Hybrid blend: 70% semantic similarity, 30% user tower — alpha scales with interaction count |
| Normal user | ≥ 5 interactions | Full two-tower retrieval + cross-encoder re-ranking |

## Phase 2 Roadmap

The current user tower uses purchase history metadata only — there is no query text encoder.
Adding `sentence-transformers/all-MiniLM-L6-v2` as a query tower with synthetic query
training pairs would enable text-driven retrieval and is the highest-impact next improvement.

Infrastructure already in place:
- 260,000 synthetic query–item pairs generated with Claude Haiku (52k items × 5 paraphrases each)
- The projection layer already accepts `concat(user_emb, query_emb)`
- The query box in the demo Phase 2 tab shows the intended UX

Expected improvement: Recall@100 ~12% → ~25–35% based on comparable systems in the literature.

## How to Run

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Run data pipeline
python data/pipeline.py

# 3. Encode item titles
python scripts/encode_item_titles.py

# 4. Train all 3 configs (requires GPU)
python model/train.py --config 1
python model/train.py --config 2
python model/train.py --config 3

# 5. Build FAISS index
python retrieval/faiss_index.py

# 6. Fine-tune cross-encoder
python retrieval/cross_encoder.py --train

# 7. Run evaluation
python experiments/run_experiment.py --config 3

# 8. Generate demo users
python scripts/generate_demo_users_v3.py

# 9. Launch demo locally
python demo/app.py
```
